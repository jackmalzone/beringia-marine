# Mindbody Documentation Organization

This document describes how the Mindbody documentation is organized.

## Directory Structure

```
docs/mindbody/
├── README.md              # Main index and navigation
├── SETUP.md               # Setup guide and configuration
├── TUTORIAL.md            # Add Client step-by-step tutorial
├── QUICK_REFERENCE.md     # Quick endpoint summary
├── ENDPOINTS_BY_FORM.md   # Endpoint mapping by form type
├── API_REFERENCE.md       # Complete API 6.0 endpoint reference
├── MODELS.md              # All data models and structures
├── IMPLEMENTATION.md      # Implementation status and requirements
├── BEST_PRACTICES.md      # API usage best practices
└── WIDGETS.md             # Widget integration reference
```

## Documentation Categories

### 🚀 Getting Started

## Navigation Flow

### For New Developers

1. Start with [README.md](./README.md) for overview
2. Read [SETUP.md](./SETUP.md) for configuration
3. Follow [TUTORIAL.md](./TUTORIAL.md) for hands-on learning
4. Reference [BEST_PRACTICES.md](./BEST_PRACTICES.md) for guidelines

### For Quick Reference

1. [QUICK_REFERENCE.md](./QUICK_REFERENCE.md) - Quick endpoint lookup
2. [ENDPOINTS_BY_FORM.md](./ENDPOINTS_BY_FORM.md) - Find endpoints for specific forms
3. [MODELS.md](./MODELS.md) - Look up data structures

### For Deep Dive

1. [API_REFERENCE.md](./API_REFERENCE.md) - Complete API documentation
2. [MODELS.md](./MODELS.md) - All data models
3. [BEST_PRACTICES.md](./BEST_PRACTICES.md) - Advanced usage patterns

## Cross-References

All documentation files include "Related Documentation" sections that link to:

- Main index (README.md)
- Related topics
- Prerequisites
- Next steps

This creates a web of documentation that's easy to navigate.

## File Naming Convention

- **UPPERCASE.md** - Main documentation files (SETUP, TUTORIAL, etc.)
- **README.md** - Index/navigation file
- **Descriptive names** - Clear purpose from filename

## Maintenance

When adding new documentation:

1. Add entry to [README.md](./README.md) index
2. Update cross-references in related files
3. Follow existing naming conventions
4. Include "Related Documentation" section
